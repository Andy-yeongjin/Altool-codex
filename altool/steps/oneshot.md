# /al oneshot — 기능 설명 → 구현 완료 전 자동 파이프라인

**사용법**: `/al oneshot [기능 설명]`

**예시**:
```
/al oneshot 사용자가 음식을 검색해서 끼니별로 기록하고 하루 칼로리를 확인하는 식단 기록 기능
```

기능 설명 하나로 개발 계획 → 상세 설계 → 구현 → 갭 분석 → 자동 개선 → 브라우저 검증까지 **6단계를 자동 완주**합니다.
각 단계는 `altool/steps/`의 해당 step을 **자동 파이프라인 모드**(Checkpoint 생략, 권장 옵션 자동 선택 — CLAUDE.md 규칙)로 실행합니다.
헌법·디자인 토큰 강제는 엔진에 내장되어 있으므로 별도 지시문 주입이 필요 없습니다.

---

## 실행 전 체크 (자동)

1. **기능 설명 확인**: 비어있으면 → 사용법 안내 후 중단.
2. **기능명 결정**: 기능 설명에서 짧은 한글 기능명을 도출한다 (공백은 하이픈).
3. **입력 자산 감지**: `🥚 [0/6] 입력 자산 감지 중...` 출력 후 탐색:

   | 자산 | 경로 | 필수 | 없을 때 |
   |------|------|------|---------|
   | 엔진 | `altool/` | 필수 | 중단 — setup.bat 안내 |
   | 헌법 | `constitution.md` | 권장 | 1회 경고 후 계속 (CLAUDE.md 자산 규칙) |
   | 디자인 시스템 | `designs/design.md` + `design-tokens.css` | 권장 | 1회 경고 후 계속 |
   | PRD | `prd/*.md` | 선택 | — 표시 |
   | PRD 참고 | `prd/refs/*` (전부 읽음, PRD 우선) | 선택 | — 표시 |
   | 디자인 화면 | `designs/*.pen` 또는 `designs/stitch/` | 선택 | — 표시 |

   PRD·.pen이 여러 개면 → 멈추고 사용자에게 선택 요청 (단, `prd/refs/`는 전부 자동 읽기).

   감지 결과 보고:
   ```
   [0/6] 입력 자산 감지 완료
     헌법: ✅/—  디자인 시스템: ✅/—  PRD: ✅/—  PRD 참고: ✅/—  디자인 화면: ✅/—
   ```

---

## 파이프라인 (6단계)

> 각 단계 시작 전 `🥚 [N/6] 단계명 시작...`, 완료 후 `🐣 [N/6] 단계명 완료` 출력 (필수).
> **멈추는 경우는 두 가지뿐**: ① 6단계에서 개발 서버 미실행, ② fix 2회 후에도 미해소 갭 잔존.

| 단계 | 실행 | 비고 |
|------|------|------|
| [1/6] 개발 계획 | `altool/steps/plan.md` 수행 | 감지된 PRD·refs·.pen을 컨텍스트로 전달 |
| [2/6] 상세 설계 | `altool/steps/design.md` 수행 | 아키텍처 Option C 자동 선택 |
| [3/6] 코드 구현 | `altool/steps/run.md` 수행 | Depth-First, 빌드 검증 포함 |
| [4/6] 갭 분석 | `altool/steps/analyze.md` 수행 | 독립 검증 에이전트 + 가능 시 런타임 검증 |
| [5/6] 자동 개선 | 미해소 갭 0건 → `🐣 [5/6] 패스` / 갭 존재 → `altool/steps/fix.md` 수행 (Match Rate ≥90%여도 갭이 있으면 수정) | **최대 2회**, 이후에도 미달 시 멈추고 선택지 제시 (A: 직접 확인 후 계속 / B: 이대로 6단계 진행) |
| [6/6] 브라우저 검증 | 아래 절차 | Playwright MCP |

### [6/6] 브라우저 기능·디자인 검증 + 자체 수정

1. `http://localhost:3000` 접속 확인 — 서버 미실행이면 **멈추고 안내**:
   ```
   ℹ️ 개발 서버가 필요합니다. start.bat(없으면 npm run dev)을 실행한 후 "계속"을 입력해주세요.
   ```
2. 주요 화면 전부 방문 + 스크린샷 수집
3. 핵심 기능 직접 테스트 (버튼 클릭, 폼 입력, 페이지 이동, 대화상자 처리)
4. `designs/design.md` 기준 디자인 검증 — 토큰 미적용·레이아웃 깨짐·375px 반응형
5. 발견된 문제는 즉시 수정 후 재검증 (설계 결함이면 설계 문서 먼저 수정 — 헌법 제9조)
6. 결과 출력: 화면 수 / 기능 검증 N개 / 디자인 검증 N개 / 수정 파일 목록
