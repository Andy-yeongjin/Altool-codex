# /al lesson — 바이브코딩 교훈 기록·조회

**사용법**:
```
/al lesson                  ← 조회 (최근 교훈 + 키워드 검색)
/al lesson {겪은 문제·교훈}   ← 기록
```

**저장소**: `~/.altool/lesson.md` **단 하나** (Windows: `$env:USERPROFILE\.altool\lesson.md`).
모든 프로젝트가 공유하는 글로벌 교훈 노트다. 파일·폴더가 없으면 헤더와 함께 생성한다.
프로젝트 고유 교훈도 여기에 적되 **태그에 프로젝트명을 포함**한다 (별도 파일 없음 — 저장소가 하나여야 헷갈리지 않는다).

---

## 조회 모드 (인자 없음)

1. `~/.altool/lesson.md`를 읽는다 (없으면 "아직 교훈이 없습니다. /al lesson {내용}으로 기록하세요" 안내 후 종료).
2. 출력: 총 건수 + **최근 5건**의 `L-번호 | 날짜 | 태그 | 예방 규칙` 한 줄 요약.
3. 검색: `/al lesson {검색어}` 입력이 기존 항목의 태그·내용과 매칭되면 기록 모드 대신 해당 항목 전문을 출력한다 (애매하면 사용자에게 "조회/기록?" 확인).

## 기록 모드 (인자 있음)

1. **중복 확인**: 같은 원인의 교훈이 이미 있으면 새 항목 대신 **기존 항목을 갱신**하고 날짜를 추가한다.
2. **항목 작성** — 4줄 형식 엄수 (장황 금지, 핵심은 예방 규칙). 프로젝트 고유 사정이면 태그에 프로젝트명 포함:

```markdown
## L-{다음번호} | YYYY-MM-DD | {태그, 쉼표 구분 — 필요 시 프로젝트명 포함}
- **증상**: {겉으로 보인 문제}
- **원인**: {진짜 원인 — 증상과 구분}
- **해결**: {실제로 통한 조치}
- **예방 규칙**: {다음에 같은 상황을 피하는 한 줄 행동 규칙}
```

3. 완료 보고:
```
🐣 [al:lesson] 기록 완료 — L-{번호} | 태그: {…}
```

---

## 자동 회고 연동 (다른 step들이 이 규칙을 따른다)

| 시점 | 동작 |
|------|------|
| **plan·run 시작 시** | `~/.altool/lesson.md`를 읽고, 이번 작업과 **관련된 교훈만** 골라 `📚 회고: L-002(인코딩), L-003(빌드)` 형식으로 1줄 출력 후 작업에 반영. 관련 없으면 출력 생략 |
| **analyze 시작 시** | 관련 교훈을 **독립 검증 에이전트에게 전달** — ① 검증 렌즈: 설계-구현이 일치해도 예방 규칙 위반이면 갭으로 보고 (Match Rate가 못 잡는 영역), ② 검증 도구 함정 회피 (인코딩·빌드 경합 등). 검사 결과는 analysis §5에 기록 |
| **fix에서 갭 수정 시** | 갭의 원인이 재발 가능한 교훈감이면 수정 완료 후 기록 **제안** (자동 기록 금지 — 사용자 확인. 단 oneshot 자동 모드에서는 기록 후 보고) |
| **report 회고(§6) 작성 시** | Problem 항목 중 재발 가능한 것을 lesson 승격 후보로 제시 |
