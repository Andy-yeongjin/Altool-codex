PART 1 · 4/5🤖 Claude에게 프롬프트

Claude에게 DB 연결 시키기

{{POOLED_URL}} 과 {{DIRECT_URL}} 두 곳만 3단계에서 복사한 값으로 교체 후 붙여넣기

🤖 Claude Code 대화창에 붙여넣기
constitution.md를 읽고 모든 원칙을 준수해줘. 특히 보안: API 키/DB 자격증명은 코드에 하드코딩 금지, .env 로만 관리, .gitignore에 .env 포함 확인. NeonDB(PostgreSQL)를 이 프로젝트에 Prisma ORM으로 연결해줘. 현재 이 프로젝트는 sqlite로 개발 중일 가능성이 높으니, 기존 sqlite 스키마와 모델 정의를 반드시 먼저 확인하고 그대로 이관해야 한다. 아래 절차를 순서대로 수행해: 0. 현재 상태 점검 (가장 먼저) - prisma/schema.prisma 가 있는지 확인하고 있으면 datasource provider, model 정의를 모두 읽어줘 - 코드 내에서 prisma.* 호출이 어떤 모델·필드를 사용하는지 grep으로 파악 - 위 두 가지를 본 뒤 이관 계획을 한 줄로 요약해서 알려줘 (어떤 모델 N개를 어떻게 옮길지) 1. 의존성 확인 - prisma (devDependency), @prisma/client 설치 여부 확인 (없으면 설치) - prisma/schema.prisma 가 이미 있으면 그걸 기준으로 작업, 없으면 npx prisma init 2. .env 파일에 아래 두 환경변수 작성 (이미 있으면 갱신): DATABASE_URL="{{POOLED_URL}}" DIRECT_URL="{{DIRECT_URL}}" 기존에 sqlite용 DATABASE_URL="file:./dev.db" 가 있었다면 위 값으로 교체. 3. prisma/schema.prisma 수정: - datasource db 블록의 provider 를 "postgresql"로 (현재 "sqlite"였다면 교체) - url을 env("DATABASE_URL")로 설정 - 같은 블록에 directUrl = env("DIRECT_URL") 추가 - 기존 model 정의는 가능한 그대로 유지. sqlite ↔ postgresql 간 타입 차이 부분만 PostgreSQL에 맞게 미세 조정 - 정의된 model이 하나도 없을 때만, 메모리/파일/JSON으로 관리되던 데이터 구조를 분석해서 새로 모델 정의 4. 기존 sqlite 마이그레이션 정리: - prisma/migrations/ 폴더가 sqlite 기준으로 만들어져 있다면 PostgreSQL과 호환되지 않으므로 폴더 통째로 삭제 - 로컬 dev.db, *.db, *.db-journal 가 .gitignore에 포함되어 있는지 확인 (없으면 추가) 5. 마이그레이션 실행: - npx prisma migrate dev --name init - npx prisma generate 6. lib/prisma.ts 싱글톤 클라이언트 생성 — Next.js dev 모드 hot reload 시 연결 풀 누수 방지 패턴 적용 (globalThis 캐싱). 이미 있으면 건너뜀. 7. 기존에 메모리/파일/sqlite로 데이터를 읽고 쓰던 부분을 모두 Prisma 클라이언트 호출로 교체. Server Actions / Route Handlers 안에서만 사용하고, 클라이언트 컴포넌트에는 절대 import하지 말 것. 8. .gitignore에 .env, .env*.local 포함 확인 (없으면 추가). 9. npm run build로 프로덕션 빌드 검증. 작업 후 어떤 파일이 추가/변경됐고, sqlite에서 가져온 모델이 무엇인지, 새로 추가한 모델이 무엇인지 요약해줘.
URL 두 개만 교체하면 됩니다. 복사 버튼으로 전체 프롬프트를 복사해 Claude Code에 붙여넣으세요.
PART 1 · 5/5 🤖 Claude에게 프롬프트

DB 연결 확인 프롬프트

🤖 위 작업 완료 후 바로 이어서 붙여넣기
방금 연결한 NeonDB가 정상 동작하는지 다음 절차로 검증하고 결과를 표로 보고해줘: 1. 연결 ping - lib/prisma.ts (또는 동일한 싱글톤)을 import한 임시 스크립트 scripts/check-db.ts 를 만들어서 await prisma.$queryRaw`SELECT 1 as ok` 실행 → 결과 출력 - tsx 또는 ts-node로 실행: npx tsx scripts/check-db.ts 2. 마이그레이션 상태 점검 - npx prisma migrate status 실행해서 "Database schema is up to date" 가 나오는지 확인 3. 모델별 row count - schema.prisma 안의 모든 model에 대해 prisma.<model>.count() 를 호출해서 | 모델명 | row count | 비고 | 형식의 표로 정리 4. 임시 INSERT/SELECT/DELETE 라운드트립 - 가장 단순한 모델 하나(예: User 같은 단일 PK 모델)를 골라 테스트용 row 1개 insert → 같은 조건으로 select 해서 일치 확인 → delete - 모델이 없으면 4번은 "스킵: 적절한 모델 없음"으로 보고 5. 종료 정리 - scripts/check-db.ts 는 작업이 끝나면 삭제 - 어떤 단계에서 어떤 출력이 나왔는지 그대로 보고 - 실패가 있다면 에러 코드(P1001/P1003/P2021 등)와 원인 한 줄 마지막에 ✅ 연결 정상 / ⚠️ 부분 정상 / ❌ 실패 중 하나로 결론 내려줘.

PART 2 · 2/6🤖 Claude에게 프롬프트

Claude에게 푸시 시키기

{{REPO_URL}} 만 10단계에서 복사한 저장소 URL로 교체 후 붙여넣기

🤖 Claude Code 대화창에 붙여넣기
constitution.md를 읽고 모든 원칙 준수. 특히 보안: .env가 .gitignore에 포함되어 있는지 반드시 먼저 확인하고, 자격증명이 한 번도 커밋된 적이 없는지 git log로 검증해줘. 이 프로젝트를 GitHub에 푸시할 거야. 아래 절차를 순서대로 수행해: 1. .gitignore 점검: - .env, .env*.local, node_modules, .next, .vercel 가 포함되어 있는지 확인 - 누락된 항목은 추가 2. .env.example 파일 생성: - 현재 .env에 있는 모든 키를 가져오되 값은 모두 빈 문자열로 - 각 키 위에 한 줄 주석으로 어디서 받는 값인지 표기 - (예: # NeonDB → Connection pooling string) 3. Vercel 배포 준비: - package.json scripts.build를 "prisma generate && next build" 로 수정 (Prisma 사용 중일 때만. Drizzle 등 다른 ORM이면 그에 맞게) - postinstall 스크립트가 필요하면 "prisma generate" 추가 - next.config.js / next.config.ts 검토 — Vercel에서 깨질 만한 설정 없는지 확인 4. README.md 갱신 (없으면 생성): - 프로젝트 한 줄 설명 - 로컬 실행 방법 (npm install → .env.example을 .env로 복사 후 값 채우기 → npm run dev) - 사용 중인 환경변수 목록과 출처 5. git 초기화 및 푸시: - 현재 디렉터리가 git 저장소가 아니면 git init - 기본 브랜치를 main으로 설정 (git branch -M main) - origin 추가: git remote add origin {{REPO_URL}} - 첫 커밋 생성 후 git push -u origin main 6. 푸시 후 git status로 작업 트리 깨끗한지 확인하고, .env 파일이 절대 트래킹되지 않은 상태인지 git ls-files로 검증해줘. 작업 종료 후 어떤 파일이 추가/변경됐는지, .env가 안전한지 한 줄로 보고해줘.
🚨
필수 확인! push 후 GitHub 저장소 새로고침 → .env 파일이 보이지 않는지 눈으로 직접 확인하세요.
.env 파일이 GitHub에 올라가면 비밀번호가 노출됩니다. 반드시 눈으로 확인하세요.
